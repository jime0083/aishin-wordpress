====================================================================
1. Dockerって、そもそも何？
====================================================================

ひとことで言うと:
  「アプリを“箱”ごと丸ごとパッケージして、どのPCでも同じように
    動かせるようにする仕組み」

■ 何がうれしいのか（Dockerがない世界の困りごと）

WordPressを自分のMacで動かすには、本来こんな準備が必要です:
  - PHP をインストール（バージョンも合わせる）
  - MySQL（データベース）をインストール・設定
  - Webサーバー（Apache）を設定
  - それぞれの設定ファイルを正しく書く
  …これを手作業でやると、途中で1つ間違えただけで動かず、
    「自分のMacでは動くのに、他の人のPCでは動かない」問題が起きがち。

Dockerを使うと:
  「PHPもMySQLもApacheも全部入った“箱”を1コマンドで起動」
  できます。あなたのMacの中に、汚さずに、隔離された小さなサーバーが
  立ち上がるイメージです。使い終わったら箱ごと捨てれば、Macには
  何も残りません（＝環境を汚さない）。

■ よく使われるたとえ

  「引っ越し用のコンテナ」
    港のコンテナは、中身が何であれ同じ規格の箱に詰めるので、
    船・トラック・クレーンがどこでも同じように運べます。
    Dockerも、アプリを標準規格の「コンテナ」に詰めることで、
    どのPC・どのサーバーでも同じように動かせます。
    （Dockerのロゴがクジラ🐳＋コンテナなのはこのため）

2. 3つの超重要キーワード（イメージ / コンテナ / ボリューム）
====================================================================

Dockerを理解するには、この3つだけ押さえればOKです。

--------------------------------------------------------------------
① イメージ（Image）＝ 箱の「設計図・型」
--------------------------------------------------------------------
  アプリと必要な部品一式が固められた「読み取り専用のテンプレート」。
  例: 「wordpress:latest」= WordPress入りの型
      「mysql:8.0」        = MySQL（DB）入りの型

  料理でたとえると「レシピ＋冷凍された食材セット」。
  イメージ自体は動きません。あくまで“型”です。

  ※イメージはインターネット上の倉庫「Docker Hub」から自動で
    ダウンロードされます（初回だけ時間がかかる理由がこれ）。

--------------------------------------------------------------------
② コンテナ（Container）＝ 型から作った「実際に動く箱」
--------------------------------------------------------------------
  イメージ（型）から起動した“実体”。これが実際に動くサーバーです。
  1つのイメージから、コンテナは何個でも作れます。

  料理でたとえると「レシピから実際に作った料理」。
  同じレシピ（イメージ）から、何皿でも作れます（コンテナ）。

  このプロジェクトでは3つのコンテナが同時に動きます:
    - wordpress コンテナ … Webサイト本体（PHP+Apache）
    - db コンテナ ………… データベース（MySQL）
    - phpmyadmin コンテナ … DBを画面で見る管理ツール

--------------------------------------------------------------------
③ ボリューム（Volume）＝ 箱の外にある「データの保管庫」
--------------------------------------------------------------------
  コンテナは「使い捨て」が基本です。消すと中のデータも消えます。
  でもDB（記事や設定）が消えたら困りますよね。

  そこで「ボリューム」という“コンテナの外の保管庫”にデータを
  置いておきます。コンテナを消して作り直しても、ボリュームに
  入れたデータは残ります。

  料理でたとえると「冷蔵庫」。
  皿（コンテナ）を洗っても、冷蔵庫（ボリューム）の食材は残る。

  このプロジェクトのボリューム:
    - db_data … データベースの中身（記事・設定など）
    - wp_data … WordPress本体のファイル

  ※テーマのコードだけは別扱い（後述の「マウント」参照）。

3. Docker Desktop と docker compose の関係
====================================================================

■ Docker Desktop（さっきインストールしたアプリ）
  Macの中でDockerを動かすための土台アプリ。
  メニューバーのクジラ🐳が出ていれば「Dockerエンジンが起動中」。
  これが動いていないと、下記のコマンドは全部エラーになります。
  → 作業前は必ずDocker Desktopを起動しておく。

■ docker compose（コンポーズ）
  「複数のコンテナをまとめて管理する道具」。

  このプロジェクトはコンテナが3つ（WordPress / DB / phpMyAdmin）。
  これを1つずつ手で起動するのは大変なので、
  「docker-compose.yml」という1枚の設計書に
  「どのコンテナを、どう繋いで、どう動かすか」を書いておき、
  1コマンド（docker compose up）で全部まとめて起動します。

  → docker-compose.yml ＝ 「3つのコンテナの合体設計図」

====================================================================
4. このプロジェクトの docker-compose.yml を読んでみる
====================================================================

ファイル: /Users/jime0083/aishin-wordpress/docker-compose.yml

ざっくり中身の意味（抜粋・かみ砕き）:

  services:              ← ここに動かすコンテナを並べる
    db:                  ← ①データベース
      image: mysql:8.0   ← MySQLの型を使う
      environment:       ← DB名やパスワードなどの初期設定（開発用の固定値）
      volumes:
        - db_data:/var/lib/mysql   ← DBの中身を保管庫db_dataに保存

    wordpress:           ← ②WordPress本体
      image: wordpress:latest
      ports:
        - "8000:80"      ← Macの8000番ポートを、箱の中の80番に繋ぐ
                            （だから http://localhost:8000 で見られる）
      volumes:
        - wp_data:/var/www/html                          ← WP本体を保管
        - ./wp-content/themes/aishin:/…/themes/aishin    ← ★重要（下記）

    phpmyadmin:          ← ③DBを画面で見る管理ツール
      ports:
        - "8080:80"      ← http://localhost:8080 で開ける

  volumes:               ← 使う保管庫の宣言
    db_data:
    wp_data:

■ ポート「8000:80」の意味（初心者がつまずきやすい点）

  「Macの8000番 : コンテナの中の80番」という“通し穴”の設定です。
  Webサーバーは箱の中では80番で動いていますが、それを
  Macの8000番につなぐことで、ブラウザで
  http://localhost:8000 と打つと箱の中のサイトが見られます。

■ 「マウント」＝ Macのフォルダを箱の中と“共有”する

  この行が超重要:
    ./wp-content/themes/aishin  :  /var/www/html/wp-content/themes/aishin
    （Mac側のテーマフォルダ）      （箱の中のテーマフォルダ）

  これは「Macのこのフォルダと、箱の中のこのフォルダを
  リアルタイムで同じにする（＝マウント）」という設定です。

  おかげで:
    Macでテーマのコード（PHP/CSS/JS）を編集すると、
    箱の中のWordPressに即座に反映される！

  → コードはMac側でGit管理しつつ、動作は箱の中のWordPressで確認、
    という理想的な開発ができる。
    （DBやWP本体は「ボリューム」に、テーマは「マウント」で、と
      使い分けているのがこの構成のポイント）

5. よく使うコマンド（このプロジェクトでの実用セット）
====================================================================

※すべて aishin-wordpress フォルダの中で実行します。
※Claude Code のプロンプトで先頭に「!」を付けると、この会話の中で
  実行できて結果を共有できます（例:  ! docker compose ps ）。

■ 起動する（3つのコンテナをまとめて立ち上げ）
    docker compose up -d
      -d は「バックグラウンドで動かす」の意味。
      初回はイメージのダウンロードで数分かかる（2回目以降は速い）。

■ 状態を見る（今どのコンテナが動いてる？）
    docker compose ps

■ ログ（中で何が起きてるか）を見る
    docker compose logs wordpress
    docker compose logs -f wordpress   ← -f でリアルタイム追尾

■ 止める（コンテナを停止・削除。データは残る）
    docker compose down

■ 完全リセット（DBもWP本体も消して最初から。テーマのコードは残る）
    docker compose down -v
      -v は「ボリュームも消す」。まっさらからやり直したい時だけ。
      ※テーマフォルダはマウントなのでMac側に残る＝コードは消えない。

■ WP-CLI（WordPressをコマンドで操作する道具）を使う
    docker compose run --rm wpcli <コマンド>
      例) docker compose run --rm wpcli core version
      --rm は「使い終わったら片付ける」の意味。

■ ブラウザで見るURL
    http://localhost:8000        … サイト本体
    http://localhost:8000/wp-admin … WordPress管理画面
    http://localhost:8080        … phpMyAdmin（DB管理）

6. 状態の全体像（頭の中のイメージ図）
====================================================================

  あなたのMac
  ├── Docker Desktop（土台・クジラ🐳が起動中）
  │     │
  │     ├─ 【コンテナ】wordpress ── ポート8000 → ブラウザで表示
  │     │        ↑ テーマフォルダをMacと共有（マウント）
  │     ├─ 【コンテナ】db (MySQL) ── 記事や設定を保存
  │     └─ 【コンテナ】phpmyadmin ── ポート8080 → DBを画面で確認
  │
  ├── 【ボリューム】db_data / wp_data（箱の外の保管庫・消えない）
  │
  └── aishin-wordpress/ フォルダ（Git管理）
        ├─ docker-compose.yml（3コンテナの設計図）
        └─ wp-content/themes/aishin/（テーマ本体＝コンテナと共有）

7. よくあるトラブルと対処
====================================================================

  | 症状 | 原因 | 対処 |
  |------|------|------|
  | コマンドが Cannot connect to the Docker daemon | Docker Desktop未起動 | クジラ🐳を起動してから再実行 |
  | http://localhost:8000 が開かない | 起動直後でまだ準備中 | 1〜2分待つ / docker compose logs で確認 |
  | ポート8000は使用中エラー | 他アプリが8000番使用 | 他を止める / ymlのポート番号を変更 |
  | 変更が反映されない | ブラウザキャッシュ | スーパーリロード（Cmd+Shift+R） |
  | DBを最初からやり直したい | — | docker compose down -v で全消し→up |

8. このあとの流れ（あなたの操作は不要・AIが代行）
====================================================================

Dockerの準備ができたので、次はAI（Claude）が以下を自動実行します:

  1. docker compose up -d ……… 環境を起動
  2. WP-CLIでWordPressを初期セットアップ（日本語・サイト名など）
  3. テーマ「aishin」を有効化
  4. ACFプラグイン導入＋シードスクリプト実行
     （インタビュー3名＋固定ページを自動投入）
  5. 全ページの表示確認（React版との照合＝Phase 6）

あなたは仕上がったサイトを http://localhost:8000 で確認するだけでOKです。

用語ミニ辞典
====================================================================
  イメージ ……… アプリの型・設計図（読み取り専用）
  コンテナ ……… 型から起動した動く実体（使い捨て）
  ボリューム …… コンテナの外にあるデータ保管庫（消えない）
  マウント ……… Macのフォルダと箱の中を共有する仕組み
  ポート ……… Macと箱の中をつなぐ通し穴（例 8000:80）
  Docker Hub … イメージが置いてあるネット上の倉庫
  compose …… 複数コンテナをまとめて管理する道具
  WP-CLI …… WordPressをコマンドで操作する道具
  デーモン …… 裏で常駐して動くプログラム（Dockerエンジン本体）
